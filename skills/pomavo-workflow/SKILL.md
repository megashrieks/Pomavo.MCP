# Pomavo Workflow Authoring

A **workflow** is a state machine: a set of **states** (nodes) and **transitions** (directed
edges between states). Templates reference a workflow; a ticket's current state is one of the
workflow's states. Workflows are **org-scoped**.

## State

```json
{
  "id": "s-created",          // GUID string; auto-generated by the granular tools if omitted
  "name": "Created",
  "description": "Newly created",
  "category": "initial",       // one of: initial | default | terminal
  "color": "#64748b",          // hex color
  "icon": "circle",            // optional lucide icon
  "graphData": "{}"            // JSON string for the visual designer (reactflow node data)
}
```

- **`category`** must be one of `initial`, `default`, `terminal`.
- **`color`** is a required hex string.
- **`graphData`** is a required JSON **string**. It stores the node's canvas data for the
  visual workflow designer (e.g. `"{\"position\":{\"x\":120,\"y\":0}}"`). `"{}"` is acceptable
  when you don't care about layout.

### The required initial state

Every valid workflow MUST have at least one state with `category: "initial"` **named exactly
`Created`**. This is the state new tickets start in. Without it, creating a template that uses
the workflow fails.

- `terminal` states represent completed/closed work (e.g. Done, Cancelled).
- `default` states are everything in between (e.g. In Progress, In Review).

## Transition

```json
{
  "name": "Start work",
  "description": "Begin working the ticket",
  "fromStateId": "s-created",   // must reference an existing state id
  "toStateId": "s-progress"      // must reference an existing state id
}
```

Transitions are **directed**: they allow moving a ticket `from` one state `to` another. Only
transitions that exist are offered when moving a ticket.

## Whole-workflow vs. granular editing

The backend stores states and transitions as complete collections. There are two ways to work:

- **Whole workflow** — `create_workflow` and `update_workflow` take the full `states` and
  `transitions` arrays. On update, any state you omit is **deleted** (and its transitions with
  it).
- **Granular (recommended for small edits)** — `add_workflow_state`, `remove_workflow_state`,
  `add_workflow_transition`, `remove_workflow_transition`. These fetch the workflow, mutate the
  one item, and save — so you don't have to re-send the whole graph.
  - `remove_workflow_state` also drops any transition that touches the removed state.
  - Identify a state by `state_id` or `name`; a transition by `transition_id` or `name`.

## Deleting

`delete_workflow` deletes the whole workflow, but **fails if any template still references it**.
Repoint those templates (`update_template_workflow`) first.

## Minimal valid workflow

```json
{
  "name": "Simple flow",
  "description": "Created -> In Progress -> Done",
  "states": [
    { "id": "s-created", "name": "Created", "description": "New", "category": "initial", "color": "#64748b", "graphData": "{}" },
    { "id": "s-prog",    "name": "In Progress", "description": "Working", "category": "default", "color": "#3b82f6", "graphData": "{}" },
    { "id": "s-done",    "name": "Done", "description": "Complete", "category": "terminal", "color": "#22c55e", "graphData": "{}" }
  ],
  "transitions": [
    { "name": "Start", "description": "Begin", "fromStateId": "s-created", "toStateId": "s-prog" },
    { "name": "Finish", "description": "Complete", "fromStateId": "s-prog", "toStateId": "s-done" }
  ]
}
```

## Typical granular session

1. `create_workflow` with just the `Created` state (+ any others you know).
2. `add_workflow_state` for each additional state (colors + categories).
3. `add_workflow_transition` to wire the allowed moves.
4. `get_workflow` to review; `remove_workflow_state` / `remove_workflow_transition` to fix.

## Authoring best practices (multiple templates)

When several templates need workflows, prefer consistency to keep boards readable (boards show
the states of **all** templates side-by-side, so divergent state sets waste horizontal space and
look messy):

1. **Reuse one workflow across templates when you can.** If the templates share the same
   lifecycle, point them all at a single workflow (`update_template_workflow`) instead of
   creating a separate near-identical workflow per template. One workflow = one clean set of
   board columns.
2. **If you genuinely need different workflows, keep state names and icons common.** When the
   use cases truly differ, still align the shared states' `name` and `icon` (and ideally
   `color`) across the workflows so the board groups/aligns them cleanly. Only let the states
   that are actually unique to a use case differ.
